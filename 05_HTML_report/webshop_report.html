Árbevételkiesés februárban — HUF 831k

A februári árbevétel HUF 831k-val csökkent 1 484-ről 652-re januárhoz képest. A napi átlagbevétel 38%-kal alacsonyabb a januári szintnél. Minden channel és termékkategória érintett a visszaesésben, operatív okokat valószínűsítünk.

A legjelentősebben a termékportfólió változás jellemzi a helyzetet. A januárban jól teljesítő termékek jelentős része nem generált semmilyen árbevételt februárban. 26 db eltűnt terméket azonosítottunk.

Valószínű ok

Operatív probléma: készlethiány, beszerzési lánc megszakadás, vagy rendszerszintű hiba. A globalitás erre utal.

Döntés: LEFT JOIN

Az orphan sorok revenue-ja az árbevétel része, még ha az ügyfél nem azonosítható is. A szegmens NULL értékét COALESCE(segment, 'UNKNOWN') kezeli.

Az INNER JOIN 2 rendelés revenue-ját veszítette volna el észrevétlenül.

Az elemzésben használt SQL

-- Feltáró lekérdezés: táblaméret

-- Orphan check: LEFT vs INNER JOIN
SELECT COUNT(*) FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id;
-- Várt: 50 (LEFT) vs 48 (INNER)

Query 04 — Revenue képlet ellenőrzés

Ellenőrizzük, hogy a revenue oszlop értékei egyeznek-e a quantity × unit_price × (1 − discount) képlettel. Ha van eltérés, az adatrögzítési hibára utalhat. Csak azokat a sorokat listázzuk, ahol az eltérés nem nulla.

SELECT order_id, quantity, unit_price, discount, revenue,
    ROUND(quantity * unit_price * (1 - discount), 2) AS szamitott,
    ROUND(revenue - quantity * unit_price * (1 - discount), 2) AS elteres
FROM orders
WHERE ROUND(revenue - quantity * unit_price * (1 - discount), 2) != 0.00
LIMIT 10;

Query 06 — Signup dátum ellenőrzés

SELECT o.order_id, o.customer_id, o.order_date, c.signup_date

3. Kapcsolatok és Orphan check

Query 07–09 — Táblaméret és JOIN count összehasonlítás

A három lekérdezés együtt dönti el, hogy LEFT JOIN-t vagy INNER JOIN-t kell használni. Ha a LEFT és INNER JOIN sorszáma eltér, vannak orphan sorok — ezek revenue-ja elveszne INNER JOIN esetén.

-- Q07: orders sorainak száma
SELECT COUNT(*) AS orders_elott FROM orders;

-- Q08: LEFT JOIN — minden orders sor megtartva
SELECT COUNT(*) AS left_join_count
FROM orders o LEFT JOIN customers c ON o.customer_id = c.customer_id;

-- Q09: INNER JOIN — csak egyező sorok
SELECT COUNT(*) AS inner_join_count
FROM orders o INNER JOIN customers c ON o.customer_id = c.customer_id;

4. Elemzési lekérdezések

Query 12 — Nettó revenue channel-enként

Az értékesítési csatornák teljesítményét hasonlítjuk össze nettó revenue, visszaküldési arány, átlagos rendelési érték (AOV) és discount szint szerint. A nettó számítás kizárja a visszaküldött rendelések bevételét.

SELECT channel,
    COUNT(*) AS orders_db,
    SUM(CASE WHEN is_returned = 'yes' THEN 1 ELSE 0 END) AS visszaru_db,
    ROUND(100.0 * SUM(CASE WHEN is_returned = 'yes' THEN 1 ELSE 0 END) / COUNT(*), 1) AS return_rate_pct,
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 2) AS netto_revenue,
    ROUND(AVG(CASE WHEN is_returned = 'no' THEN revenue END), 2) AS aov,
    ROUND(AVG(discount), 3) AS atlag_discount
FROM orders
GROUP BY channel
ORDER BY netto_revenue DESC;

Query 13c–e — Marketplace-Clothing anomália részletezés

A mátrix alapján kiemelkedő marketplace-Clothing kombinációt háromféle szempont szerint vizsgáljuk: egyedi rendelések (13c), összesített elveszett revenue (13d), és az elveszett revenue teljes nettó revenue-hoz viszonyított aránya (13e).

-- Q13c: Egyedi rendelések listája
SELECT order_id, product_name, quantity, unit_price, discount, is_returned, revenue
FROM orders
WHERE channel = 'marketplace' AND category = 'Clothing'
ORDER BY revenue DESC;

-- Q13d: Összesített elveszett revenue
SELECT ROUND(SUM(revenue), 2) AS brutto_revenue,
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 2) AS netto_revenue,
    ROUND(SUM(CASE WHEN is_returned = 'yes' THEN revenue ELSE 0 END), 2) AS elveszett_revenue,
    COUNT(*) AS osszes_order,
    SUM(CASE WHEN is_returned = 'yes' THEN 1 ELSE 0 END) AS visszaru_db
FROM orders
WHERE channel = 'marketplace' AND category = 'Clothing';

-- Q13e: Elveszett revenue % a teljes nettó revenue-hoz
SELECT
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 2) AS teljes_netto_revenue,
    ROUND(SUM(CASE WHEN channel = 'marketplace' AND category = 'Clothing'
        AND is_returned = 'yes' THEN revenue ELSE 0 END), 2) AS elveszett_revenue,
    ROUND(100.0 * SUM(CASE WHEN channel = 'marketplace' AND category = 'Clothing'
        AND is_returned = 'yes' THEN revenue ELSE 0 END) /
        SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 1) AS elveszett_pct
FROM orders;

Query 14 — Revenue szegmens szerint (LEFT JOIN)

Az ügyféltábla szegmens mezője (new / returning) alapján hasonlítjuk össze a két csoport revenue-ját. LEFT JOIN-t használunk, hogy az orphan rendelések UNKNOWN szegmensként ne vesszenek el. A COALESCE gondoskodik az ismeretlen szegmens kezeléséről.

SELECT COALESCE(c.segment, 'UNKNOWN') AS segment,
    COUNT(*) AS orders_db,
    ROUND(SUM(CASE WHEN o.is_returned = 'no' THEN o.revenue ELSE 0 END), 2) AS netto_revenue,
    ROUND(AVG(CASE WHEN o.is_returned = 'no' THEN o.revenue END), 2) AS aov
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
GROUP BY c.segment
ORDER BY netto_revenue DESC;

Query 15 — Havi trend és napi átlag

A két hónap összehasonlítása abszolút és napi átlag szinten. A napi átlag korrigál az eltérő időszakhosszra — február 19 napot fed le januárhoz képest (27 nap), ezért a napi átlag a valódi teljesítménycsökkenés mértéke.

-- Q15: Havi összesítő
SELECT strftime('%Y-%m', order_date) AS honap,
    COUNT(*) AS orders_db,
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 2) AS netto_revenue
FROM orders GROUP BY honap ORDER BY honap;

-- Q15b: Napi átlagos revenue (időszak-hossz korrekcióval)
SELECT strftime('%Y-%m', order_date) AS honap,
    COUNT(DISTINCT order_date) AS aktiv_napok,
    COUNT(*) AS orders_db,
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 2) AS netto_revenue,
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END) / COUNT(DISTINCT order_date), 2) AS napi_atlag_revenue
FROM orders GROUP BY honap ORDER BY honap;

Query 15c–f — Havi mix és hiányzó termékek feltárása

Februárban más channel-category kombináció volt aktív, mint januárban. Ezek a lekérdezések feltárják, hogy a visszaesés mögött konkrétan melyik kombinációk tűntek el, és melyek tűntek fel újonnan.

-- Q15c: Februári revenue mix channel-category bontásban
SELECT channel, category, COUNT(*) AS orders_db,
    ROUND(SUM(CASE WHEN is_returned = 'no' THEN revenue ELSE 0 END), 2) AS netto_revenue
FROM orders
WHERE strftime('%Y-%m', order_date) = '2024-02'
GROUP BY channel, category ORDER BY netto_revenue DESC;

-- Q15e: Channel-category pivot: Jan vs Feb variance
SELECT channel || ' | ' || category AS channel_category,
    ROUND(SUM(CASE WHEN strftime('%Y-%m', order_date) = '2024-01'
        AND is_returned = 'no' THEN revenue ELSE 0 END), 2) AS jan,
    ROUND(SUM(CASE WHEN strftime('%Y-%m', order_date) = '2024-02'
        AND is_returned = 'no' THEN revenue ELSE 0 END), 2) AS feb,
    feb - jan AS variance
FROM orders
GROUP BY channel, category ORDER BY variance ASC;

Query 16b — Kategória szintű variance

A termékszintű variance kategóriákra aggregálva megmutatja, hogy a visszaesés egyenletesen érinti-e a kategóriákat, vagy van-e domináns kategória. Ha mindhárom kategória érintett, az rendszerszintű problémára utal.

SELECT category,
    ROUND(SUM(CASE WHEN strftime('%Y-%m', order_date) = '2024-01'
        AND is_returned = 'no' THEN revenue ELSE 0 END), 2) AS jan_revenue,
    ROUND(SUM(CASE WHEN strftime('%Y-%m', order_date) = '2024-02'
        AND is_returned = 'no' THEN revenue ELSE 0 END), 2) AS feb_revenue,
    feb_revenue - jan_revenue AS variance
FROM orders
GROUP BY category
ORDER BY variance ASC;
